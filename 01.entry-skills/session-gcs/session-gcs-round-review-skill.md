# 轮次评价 · 追问提示词生成技能

## 功能概述

每轮模型对话结束后执行本技能。核心目标是：

1. 读取本轮对话内容（模型回答）
2. 调用 `implementation-reviewer` agent 评估本轮回答质量
3. 调用 `humanizer-zh` agent 对评价文字去 AI 化
4. 判断是否需要继续追问：
   - **满意** → 写入轮次评价，标记本轮通过，流程结束
   - **不满意** → 写入轮次评价 + 生成下一轮追问提示词，追加到对话内容文件的「用户第 N 次提示词」字段
5. 将评价内容追加到 `-评价结果.md`

**本技能不负责：**

- 替代用户在 Trae 中执行对话
- 自动提交最终评价汇总表单

## 命令

| 命令 | 说明 |
|------|------|
| review | 默认命令。分析指定模型的本轮对话，输出评价 + 追问决策 |

## 默认配置

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 对话内容文件 | `A-<项目名>-<类型>-<模型名>-对话内容.md` | 读取本轮模型回答，写入追问提示词 |
| 评价结果文件 | `A-<项目名>-<类型>-<模型名>-评价结果.md` | 追加本轮评价内容 |
| 最大追问轮次 | 3 轮（含首轮共 3 次） | 第 3 轮结束后不再生成追问 |
| 模型名映射 | original / dyad / trinity | 对应 `-对话内容.md` 文件后缀 |

## 执行流程

### review（默认命令）

**输入：**
```
项目名：<项目名>
类型：<类型>
模型：<original / dyad / trinity>
轮次：<1 / 2 / 3>
```

**步骤：**

1. **读取本轮对话内容**：从 `-对话内容.md` 中提取：
   - 本轮用户提示词（「用户第 N 次提示词」字段）
   - 本轮模型回答（「模型第 N 次回答内容」字段）
   - 本轮 session id

2. **调用 `implementation-reviewer` agent**：传入以下上下文：
   - 首轮提示词（含约束标签，作为参照基准）
   - 本轮用户提示词
   - 本轮模型回答内容
   - 已有轮次（判断对话走向）
   - 任务类型

   分析输出：
   - 本轮回答是否准确、完整、符合约束
   - 满意度判断：**满意 / 不满意**
   - 若不满意：指出具体不足（最多 2 点）及追问方向

3. **调用 `humanizer-zh` agent**：对 implementation-reviewer 输出的**所有评价文字**进行去 AI 化，包括：
   - 满意度判断理由
   - 不足描述
   - 追问提示词（若需要）

   **implementation-reviewer 原始输出禁止直接落盘，必须先经 humanizer-zh 处理。**

4. **生成追问提示词（满意度为"不满意"且未达最大轮次时）**：
   - 追问提示词应针对本轮的具体不足，自然口语化，像真实用户的跟进追问
   - 不重复首轮已问过的内容
   - 若已达第 3 轮，不生成追问，提示用户本轮为最后一轮

5. **追加评价到 `-评价结果.md`**：在对应轮次区块（`## 第 N 轮评价`）中填入：
   - 满意度：满意 / 不满意
   - 不足描述（不满意时）
   - 追问方向（不满意时）
   - 评价备注（humanizer-zh 去 AI 化后的综合评语）

6. **将追问提示词写入 `-对话内容.md`**（仅不满意时）：
   - 写入「用户第 N+1 次提示词」字段
   - 若该字段已有内容，**不覆盖**，提示用户手动确认

7. 输出操作摘要，提示用户下一步：
   - 满意：可提交本模型的评价，或等待三模型全部完成后运行 `session-gcs-analysis analyze`
   - 不满意：追问提示词已写入，可复制到 Trae 继续下一轮对话

## 路径规则

```
# 读取 + 写入追问提示词
02.work session/session-gcs/ai-model-result/<项目名>/<项目名>-<类型>/A-<项目名>-<类型>-<模型名>-对话内容.md

# 追加评价内容
02.work session/session-gcs/ai-model-result/<项目名>/<项目名>-<类型>/A-<项目名>-<类型>-<模型名>-评价结果.md
```

## 追加内容格式

每次 review 执行后，向 `-评价结果.md` 对应轮次区块追加以下内容：

```markdown
## 第 <N> 轮评价

满意度：<满意 / 不满意>

不足描述：<具体不足（humanizer-zh 去 AI 化后），满意时留空>

追问方向：<追问切入点（humanizer-zh 去 AI 化后），满意时留空>

评价备注：<综合评语（humanizer-zh 去 AI 化后）>
```

若该区块已存在内容，**不覆盖**，在末尾追加并加分隔线 `---`。

## 对话内容文件追问写入格式

当不满意且需要追问时，将追问提示词写入 `-对话内容.md` 的下一轮字段：

```markdown
用户第 <N+1> 次提示词：<humanizer-zh 优化后的追问提示词>
```

## 判断逻辑

| 条件 | 操作 |
|------|------|
| 满意，轮次 < 3 | 写评价，结束本模型本轮，等待用户决定是否继续 |
| 满意，轮次 = 3 | 写评价，提示三轮已完成 |
| 不满意，轮次 < 3 | 写评价 + 生成追问提示词，写入下一轮字段 |
| 不满意，轮次 = 3 | 写评价（标注已达最大轮次），**不生成追问** |

## 注意事项

1. **所有评价文字必须经 `humanizer-zh` 去 AI 化**，包括追问提示词，禁止直接使用 implementation-reviewer 原始输出。
2. 若 `-对话内容.md` 中本轮「模型回答内容」字段为空，说明用户尚未完成对话，**中止执行**并提示用户先填写模型回答。
3. 追问提示词要针对本轮具体不足，不能重复首轮问题，不能泛泛追问"能否详细说明"。
4. 三个模型（original / dyad / trinity）各自独立执行，互不影响。
5. 执行前先确认轮次参数，避免写入错误区块。

## 示例

### 输入

```text
项目名：label-01223
类型：代码理解
模型：dyad
轮次：1
```

### 输出

```text
本轮评价已写入：
  02.work session/session-gcs/ai-model-result/label-01223/label-01223-代码理解/A-label-01223-代码理解-dyad-评价结果.md

满意度：不满意
不足：解释了 row++ 的作用，但没结合 for 循环的整体递减结构说明为什么这样设计，略显割裂。

追问提示词已写入对话内容文件「用户第二次提示词」字段：
  "你上面说 row++ 是为了重新检查当前行，但我还是没理解——如果删掉这行，for 循环下次迭代会去看哪行？能结合这个 for 的递减结构说一下吗？"

下一步：复制追问提示词到 Trae 继续第二轮对话，完成后再次执行 review 轮次:2。
```
