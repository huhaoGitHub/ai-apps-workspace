# 分析技能

## 功能概述

这个技能用于分析 session-4 项目的模型执行结果是否完成了 prompt 要求。

它负责：

- 对照 prompt、代码和结果文件判断任务是否完成
- 区分“没做成”和“做得一般”
- 输出评价结果文件

它不负责：

- 继续实现功能
- 修复代码
- 帮模型补完未完成部分

## 执行前提

执行前必须先向用户确认以下信息：

1. **项目名**，例如：`label-01968-plus`
2. **提示词编号**，即文件名中的 `<类型>-<index>` 部分，例如：`bug修复-01`、`0-1代码生成-04`
3. **第几次对话**，例如：第 1 次、第 2 次、第 3 次

没有项目名或提示词编号时，不要自行猜测，不要直接开始分析。

如果用户只给了项目名，先列出 `02.work session/session-4/ai-model-result/<项目名>/` 下所有类型目录及文件，供用户选择。

如果用户给了项目名和编号，先读取对应提示词文件，列出其中存在内容的对话轮次（「用户第X次提示词」非空的），再询问用户要分析哪一次。

## 输入来源

拿到项目名、提示词编号和对话轮次后，按下面顺序读取上下文：

1. `02.work session/session-4/ai-model-result/<项目名>/<项目名>-<类型>/A-<项目名>-<类型>-<index>.md`——从中找到「用户第 N 次提示词」作为本次分析的 prompt，以及对应的「模型第 N 次回答内容」和「模型第 N 次回答 trae session id」
2. `02.work session/session-4/gitlab source/<项目名>-plus/<项目名>-<类型>/A-<项目名>-<类型>-<index>/`——确认该轮次对应子仓的实际代码状态

**文件与仓库必须严格对应**：提示词文件和代码子仓的 `<项目名>`、`<类型>`、`<index>` 三段必须完全一致。分析 `A-label-01035-代码测试-13.md` 时，只能读取 `label-01035-代码测试/A-label-01035-代码测试-13/` 这个子仓，不能读取其他任何子仓或主仓。读取前必须先确认路径中的 index 与当前任务完全匹配。

## 分析目标

你需要回答三个问题：

1. 模型是否完成了 prompt 中明确提出的要求
2. 这次结果整体是否满意
3. 模型在执行过程中，哪些地方做得不好，或者该扩展却没有扩展

### 评价维度（按重要性排序，工程师视角）

评价以 `implementation-reviewer` agent 的输出结果为基础依据，结合本技能的维度标准做最终判断。作为资深全栈工程师（10年+经验），评价的核心不是"能否编译验证"，而是：

1. **Prompt 理解度** (30%) — 模型是否正确理解了需求的本质，包括显式和隐式要求？
2. **实现逻辑完整性** (30%) — 代码逻辑是否正确、边界是否考虑周全（通过代码审查判断）？
3. **会话反馈响应质量** (20%) — 对用户反馈的理解是否准确、调整方向是否正确？
4. **跨迭代协调性** (15%) — 相邻迭代改动是否会产生冲突、系统整体一致性如何？
5. **代码架构质量** (5%) — 设计决定是否合理、代码风格是否一致？

**关键原则**：不因编译器缺失、环境搭建困难等不可控因素否定实现。通过代码审查和逻辑分析评价质量。

### 1）任务完成度

这里只判断“该做的事情有没有做成”。输出值只能是：

- 已完成
- 未完成

如果是“未完成”，未完成说明必须只写 prompt 中要求实现但最后没有做到的部分，不能混入体验、质量或扩展性评价。

### 2）是否满意

这里只判断对这次结果整体是否满意。输出值只能是：

- 满意
- 不满意

如果是“不满意”，必须附带“不满意的点”。

**什么时候标记满意**：Prompt 理解准确、实现逻辑正确、边界被识别（即使未全部验证）、会话反馈被有效处理、跨迭代无冲突、代码风格合理。

**什么时候标记不满意**：误解需求、代码逻辑有缺陷、反馈应对不当、跨迭代冲突、设计不合理。**不因验证失败而判不满意**。

### 评价策略

- 边界条件是否遗漏（空值、越界、并发、重复操作）
- 隐式需求是否识别（如 prompt 说"保持现有风格"，有没有真的看现有代码对比）
- 不该省略的校验是否缺失（参数校验、权限校验、状态前置条件）
- 设计决定是否合理（是否引入不必要的复杂度、是否有更简洁的方案）
- 表达是否冗余（重复解释、与 prompt 无关的大段分析）
- 跨迭代副作用是否考虑（改了 A 会不会影响 B）

### 3）过程满意度

这里评价**执行过程的质量**，包括：

- **理解和准备**：prompt 理解是否全面、信息收集是否充分？
- **处理方式**：代码改动的策略、代码审查流程是否周全？
- **反馈响应**：对用户反馈的理解、调整方向是否恰当、是否过度自我否定？
- **质量考虑**：是否主动识别隐藏风险、跨迭代协调是否前瞻性？
- **表达方式**：解说是否简洁准确、是否避免篇幅冗余？
- **验证建议**：虽然环境限制无法编译，后续如何补验证？（作为建议，不影响当前满意度）

不要把“功能根本没实现”这种内容写到过程满意度里。

## 写作要求

### 任务完成度

- 必须明确写“已完成”或“未完成”
- 如果写“未完成”，必须附带“未完成说明”

### 未完成说明

- 只写 prompt 中要求实现但最终没做到的内容
- 不能把实现效果不佳、体验一般、扩展不足混进来
- 字数控制在 100 到 200 字
- 尽量结合实际业务、实际操作过程和实际结果
- 最终成稿前，必须调用 `humanizer-zh.agent.md` 去 AI 味

### 是否满意

- 必须明确写“满意”或“不满意”
- 如果写“不满意”，必须附带“不满意的点”

### 不满意的点

- 只说具体问题，不泛泛而谈
- 按优先级写（逻辑缺陷 → 设计不合理 → 反馈响应错误 → 跨迭代冲突 → 表达拖沓）
- 不能把"根本没做成"的内容混进来，未完成内容应放在"未完成说明"
- **不写**"没有编译验证"、"环境跑不起来"这类理由，应放在过程满意度的验证建议部分
- 字数控制在 80 到 150 字
- **写法要求**：说人话，像工程师在 code review 里直接指出问题，不用"值得注意的是""此外""综上所述"这类模板句

### 过程满意度

- 重点评价**处理质量**而非验证完整性
- 可以结合代码改动、反馈响应、设计决定来写，不需要按维度逐条列举
- 写连贯的叙述段落，不要用 [理解][处理][响应] 等标签拆成条目
- 可以提验证相关建议，但不影响满意度判断
- 字数控制在 80 到 160 字
- **写法要求**：语气直接，像有经验的工程师在做 peer review，不用"综上所述""总体来看""值得肯定的是"这类套话

### 下一轮推荐提示词

仅当是否满意为"不满意"时生成。

- 基于"不满意的点"，生成一条面向模型的下一轮提示词
- 目标是让模型重新处理失败的部分，而不是重复原始 prompt
- 提示词必须聚焦一个问题，不要把所有不满意的点都塞进去——选最核心的那一个
- 如果是"方向走错"类问题（如搜索了错误目标），直接在提示词里把正确方向给出来，避免模型在下一轮继续走偏
- 如果是"没执行"类问题（如分析对但没改代码），直接在提示词里要求执行，不要再次请求分析
- 如果是"逻辑缺陷"类问题，提示词里描述的是预期行为或当前的错误现象，让模型自己定位和修
- 字数控制在 50 到 120 字，口吻贴近真实工作场景，不要写得像考试题

## 输出文件

评价结果固定追加写入与提示词文件同目录的评价结果文件：

- `02.work session/session-4/ai-model-result/<项目名>/<项目名>-<类型>/A-<项目名>-<类型>-<index>-评价结果.md`

例如：

- `02.work session/session-4/ai-model-result/label-01968-plus/label-01968-plus-bug修复/A-label-01968-plus-bug修复-01-评价结果.md`

**每次分析结果追加到文件末尾，不覆盖已有内容。**文件不存在时新建。

## 输出格式

每次追加一个完整评价块，**第一行必须是标志行**，标志从评价结果文件名中提取：取 `label-XXXX` 中的数字（去掉前导零）和 `<index>`，格式为 `A-<数字>-<index>`。例如文件名 `A-label-01035-代码测试-13-评价结果.md`，标志为 `A-1035-13`。

标志行之后是标题（带轮次编号），**标题之后必须紧跟 `## 提示词`、`## Session ID` 和 `## 文件修改` 三节**，前两节内容从对应提示词文件的第 N 次对话轮次中读取，第三节通过扫描该轮次的 `edit_file_search_replace` 和 `Write` 工具调用记录得出：

```markdown

# <项目名>-<类型>-<index> 第 N 次对话评价结果

### 标志：
A-1035-13

## Session ID
<从提示词文件中读取"模型第 N 次回答 trae session id"的完整内容>

## 提示词
<从提示词文件中读取"用户第 N 次提示词"的完整内容>

## 文件修改
单文件修改 / 多文件修改 / 无代码修改
修改文件：<文件名1>、<文件名2>（无修改时省略此行）

## 任务完成度
已完成 / 未完成

## 未完成说明
<仅当任务完成度为未完成时填写>

## 是否满意
满意 / 不满意

## 不满意的点
<仅当是否满意为不满意时填写，100-200 字>

## 过程满意度
<包括理解/处理/响应/前瞻/表达/建议等维度，100-200 字>

## 下一轮推荐提示词
<仅当是否满意为不满意时填写，50-120 字>
```

如果任务完成度为"已完成"，可以省略"未完成说明"整段。

如果是否满意为"满意"，可以省略"不满意的点"整段。

如果是否满意为"不满意"，必须追加"下一轮推荐提示词"整段。

```markdown
## 下一轮推荐提示词
<基于不满意的点，生成下一轮对话的推荐提示词>
```

## 执行流程

1. 先向用户询问项目名。
2. 列出 `02.work session/session-4/ai-model-result/<项目名>/` 下所有提示词文件，供用户选择提示词编号（类型 + index）。
3. **必须主动调用工具读取**对应提示词文件 `A-<项目名>-<类型>-<index>.md` 的最新内容，不得依赖对话上下文中已存在的附件或缓存内容——文件可能在附件展示后被用户更新。
4. **提示词文件可能非常长（超过1000行），不能从固定行号偏移读取来判断轮次是否有内容。** 必须先用 grep_search 搜索 `用户第.*次提示词` 获取所有轮次标记的行号，再根据行号范围判断每个轮次是否有实际内容（即标记行的下一行到下一个轮次标记之间有非空文本）。从读取结果中列出存在内容的对话轮次，向用户确认要分析第几次。
5. 确定轮次后，用 grep_search 定位「用户第 N 次提示词」所在行号，从该行开始读取到「用户第 N+1 次提示词」行号之前，提取 prompt 和模型回答内容。
6. 扫描该轮次内容，统计 `edit_file_search_replace` 和 `Write` 工具调用中出现的 `filePath`，去重后记录修改文件列表；0 个文件为"无代码修改"，1 个为"单文件修改"，2 个及以上为"多文件修改"。
7. 读取 `02.work session/session-4/gitlab source/<项目名>/<项目名>-<类型>/A-<项目名>-<类型>-<index>/`，确认该子仓实际代码和结果状态。
8. **调用 `implementation-reviewer` agent**，将 prompt、模型回答内容、代码子仓内容作为输入，获取其对该轮次实现质量的评价结果，作为后续判断的基础依据。
9. 对照 prompt 要求与实际结果（结合 `implementation-reviewer` 的评价），判断任务完成度。
10. 判断这次结果整体是否满意。
11. 若为未完成，撰写未完成说明。
12. 若为不满意，撰写不满意的点。
13. 撰写过程满意度。
14. **对"未完成说明"、"不满意的点"、"过程满意度"、 "下一轮推荐提示词"三节全部调用 `humanizer-zh.agent.md` 做去 AI 味润色**，然后再写入文件, 这点非常重要
15. 若是否满意为"不满意"，基于"不满意的点"生成"下一轮推荐提示词"，写入评价块末尾。
16. 将结果写入 `A-<项目名>-<类型>-<index>-评价结果.md`。

## 注意事项

1. 不要只根据模型自述判断是否完成，必须结合代码和结果文件一起看。
2. 不要把“没做成”和“做得不够好”混成一类。
3. 不要写太泛的总结，尽量落到实际代码逻辑、设计决定和会话交互。
4. 这个技能只做分析，不负责补代码、不负责修复、不负责继续实现。
5. **分析上下文来源只有一个**：`ai-model-result/<项目名>/<项目名>-<类型>/A-<项目名>-<类型>-<index>.md`。用户问"找的是哪个文件"时，指的是这个提示词文件，而不是模型在回答内容中通过工具调用读过的代码文件。两者不要混淆。
6. **每次分析前必须主动读取文件**，不得依赖对话上下文中的附件缓存或之前读取的内容。即使文件内容已经出现在上下文中，也必须重新调用工具读取，因为文件可能在附件展示后已被用户修改。
7. **不过度依赖验证环境作为评价因素**：
   - ✅ 通过代码审查判断逻辑是否正确（如 `cin.fail()` 对 `char` 无效）
   - ✅ 通过代码设计评价架构质量（如菜单键风格、子菜单设计）
   - ✅ 通过会话流程评价反馈响应质量
   - ❌ 不因编译器缺失、环境搭建困难而否定实现
   - ❌ 不把"无法验证"作为"不满意"的主要理由
   - 📝 可在"过程满意度"的"验证建议"中提及环境限制，但作为补充而非否定
8. **识别设计问题优先于验证缺失**：
   - 例：子菜单设计不合理 → 从代码结构就能评价，无需编译
   - 例：`cin.fail()` 对 `char` 无效 → 从代码逻辑就能判断，无需运行
   - 例：'a'/'A' 菜单键冲突 → 对比两份代码就能发现，无需测试
   - 这些问题不需要编译就能识别，不应因"无法验证"而忽视
9. **文件与仓库必须严格一一对应，不得错配**：
   - 提示词文件路径中的 `<项目名>`、`<类型>`、`<index>` 三段，必须与代码子仓路径完全一致
   - 分析 `A-label-01035-代码测试-13.md` 时，只能对应读取 `gitlab source/label-01035-plus/label-01035-代码测试/A-label-01035-代码测试-13/`，不得读取其他 index 的子仓或主仓
   - 读取代码前，先把要读取的子仓路径完整写出来，和提示词文件名中的标识串逐段比对，确认三段全部匹配后再继续
10. 输出一定要全部调用 `humanizer-zh.agent.md` 去 AI 话
