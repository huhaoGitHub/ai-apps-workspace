# 分析技能

## 功能概述

这个技能用于分析 session-4 项目的模型执行结果是否完成了 prompt 要求。

它负责：

- 对照 prompt、代码和结果文件判断任务是否完成
- 区分“没做成”和“做得一般”
- 输出评价结果文件

它不负责：

- 继续实现功能
- 修复代码
- 帮模型补完未完成部分

## 执行前提

执行前必须先向用户确认项目名，例如：

- `label-02404`
- `label-01849`

没有项目名时，不要自行猜测，不要直接开始分析。

## 输入来源

拿到项目名后，按下面顺序读取上下文：

1. `02.work session/session-4/gitlab source/<项目名>/`
2. `03.trae prompts/session-4.md`
3. `02.work session/session-4/ai-model-result/<项目名>.md`

如果 `03.trae prompts/session-4.md` 中同一个项目存在多条记录：

- 优先使用和当前分析目标最匹配的记录
- 如果无法明确判断，则优先使用最后一条同项目记录
- 不要把多条不同任务类型的 prompt 混在一起做一次评价

## 分析目标

你需要回答两个问题：

1. 模型是否完成了 prompt 中明确提出的要求
2. 模型在执行过程中，哪些地方做得不好，或者该扩展却没有扩展

### 1）任务完成度

这里只判断“该做的事情有没有做成”。输出值只能是：

- 已完成
- 未完成

如果是“未完成”，未完成说明必须只写 prompt 中要求实现但最后没有做到的部分，不能混入体验、质量或扩展性评价。

### 2）过程满意度

这里只写执行过程中的不足，例如：

- 虽然完成了主流程，但实现效果一般
- 边界情况没有覆盖到
- 验证过程不扎实
- 本来应该顺手补的说明或文档没有补

不要把“功能根本没实现”这种内容写到过程满意度里。

## 写作要求

### 任务完成度

- 必须明确写“已完成”或“未完成”
- 如果写“未完成”，必须附带“未完成说明”

### 未完成说明

- 只写 prompt 中要求实现但最终没做到的内容
- 不能把实现效果不佳、体验一般、扩展不足混进来
- 字数控制在 100 到 200 字
- 尽量结合实际业务、实际操作过程和实际结果
- 最终成稿前，必须调用 `humanizer-zh.agent.md` 去 AI 味

### 过程满意度

- 只写过程中的不足、效果一般之处、该扩展却没扩展的地方
- 可以结合业务场景、代码改动、验证过程来写
- 如果存在多轮对话，可以按轮次归纳不足，再给总体评价
- 字数控制在 100 到 200 字

## 输出文件

评价结果固定写入：

- `02.work session/session-4/ai-model-result/<项目名>-评价结果.md`

例如：

- `02.work session/session-4/ai-model-result/label-02404-评价结果.md`

如果文件已存在，默认覆盖旧结果，写入本次最新分析。

## 输出格式

推荐格式：

```markdown
# <项目名> 评价结果

## 任务完成度
已完成 / 未完成

## 未完成说明
<仅当任务完成度为未完成时填写>

## 过程满意度
<100 到 230 字>
```

如果任务完成度为“已完成”，可以省略“未完成说明”整段。

## 执行流程

1. 先向用户询问项目名。
2. 读取 `03.trae prompts/session-4.md`，找到本次分析对应的 prompt。
3. 读取 `02.work session/session-4/gitlab source/<项目名>/`，确认实际代码和结果状态。
4. 读取 `02.work session/session-4/ai-model-result/<项目名>.md`，整理模型执行过程和最终结果。
5. 对照 prompt 要求与实际结果，判断任务完成度。
6. 若为未完成，撰写未完成说明。
7. 调用 `humanizer-zh.agent.md` 对未完成说明做去 AI 味润色。
8. 撰写过程满意度。
9. 将结果写入评价结果文件。

## 注意事项

1. 不要只根据模型自述判断是否完成，必须结合代码和结果文件一起看。
2. 不要把“没做成”和“做得不够好”混成一类。
3. 不要写太泛的总结，尽量落到实际业务、实际功能和实际验证结果。
4. 这个技能只做分析，不负责补代码、不负责修复、不负责继续实现。
