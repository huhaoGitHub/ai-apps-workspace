# 分析技能

## 功能概述

这个技能用于分析 session-4 项目的模型执行结果是否完成了 prompt 要求。

它负责：

- 对照 prompt、代码和结果文件判断任务是否完成
- 区分“没做成”和“做得一般”
- 输出评价结果文件

它不负责：

- 继续实现功能
- 修复代码
- 帮模型补完未完成部分

## 执行前提

执行前必须先向用户确认两项信息：

1. **项目名**，例如：`label-02404`、`label-01849`
2. **第几次对话**，例如：第 1 次、第 2 次

没有项目名或对话轮次时，不要自行猜测，不要直接开始分析。

如果用户只给了项目名，先读取 `02.work session/session-4/ai-model-result/<项目名>.md`，列出文件中所有对话轮次（即所有"用户第X次提示词"），再询问用户要分析哪一次。

## 输入来源

拿到项目名和对话轮次后，按下面顺序读取上下文：

1. `02.work session/session-4/ai-model-result/<项目名>.md`——从中找到"用户第 N 次提示词"作为本次分析的 prompt，以及对应的"模型第 N 次回答内容"作为执行结果
2. `02.work session/session-4/gitlab source/<项目名>/`——确认实际代码状态

**不再**从 `03.trae prompts/session-4.md` 中查找提示词。

## 分析目标

你需要回答三个问题：

1. 模型是否完成了 prompt 中明确提出的要求
2. 这次结果整体是否满意
3. 模型在执行过程中，哪些地方做得不好，或者该扩展却没有扩展

### 1）任务完成度

这里只判断“该做的事情有没有做成”。输出值只能是：

- 已完成
- 未完成

如果是“未完成”，未完成说明必须只写 prompt 中要求实现但最后没有做到的部分，不能混入体验、质量或扩展性评价。

### 2）是否满意

这里只判断对这次结果整体是否满意。输出值只能是：

- 满意
- 不满意

如果是“不满意”，必须附带“不满意的点”。

### 3）过程满意度

这里只写执行过程中的不足，例如：

- 虽然完成了主流程，但实现效果一般
- 边界情况没有覆盖到
- 验证过程不扎实
- 本来应该顺手补的说明或文档没有补

不要把“功能根本没实现”这种内容写到过程满意度里。

## 写作要求

### 任务完成度

- 必须明确写“已完成”或“未完成”
- 如果写“未完成”，必须附带“未完成说明”

### 未完成说明

- 只写 prompt 中要求实现但最终没做到的内容
- 不能把实现效果不佳、体验一般、扩展不足混进来
- 字数控制在 100 到 200 字
- 尽量结合实际业务、实际操作过程和实际结果
- 最终成稿前，必须调用 `humanizer-zh.agent.md` 去 AI 味

### 是否满意

- 必须明确写“满意”或“不满意”
- 如果写“不满意”，必须附带“不满意的点”

### 不满意的点

- 只写让人不满意的具体问题，不要泛泛而谈
- 优先写 prompt 已要求但做得不到位、验证不充分、扩展明显不足的地方
- 不能把“根本没做成”的内容重复塞进这里，未完成内容应放在“未完成说明”
- 字数控制在 80 到 180 字

### 过程满意度

- 只写过程中的不足、效果一般之处、该扩展却没扩展的地方
- 可以结合业务场景、代码改动、验证过程来写
- 如果存在多轮对话，可以按轮次归纳不足，再给总体评价
- 字数控制在 100 到 200 字

## 输出文件

评价结果固定追加写入：

- `02.work session/session-4/ai-model-result/<项目名>-评价结果.md`

例如：

- `02.work session/session-4/ai-model-result/label-02404-评价结果.md`

**每次分析结果追加到文件末尾，不覆盖已有内容。**文件不存在时新建。

## 输出格式

每次追加一个完整评价块，标题带轮次编号：

```markdown
# <项目名> 第 N 次对话评价结果

## 任务完成度
已完成 / 未完成

## 未完成说明
<仅当任务完成度为未完成时填写>

## 是否满意
满意 / 不满意

## 不满意的点
<仅当是否满意为不满意时填写>

## 过程满意度
<100 到 230 字>
```

如果任务完成度为“已完成”，可以省略“未完成说明”整段。

如果是否满意为“满意”，可以省略“不满意的点”整段。

## 执行流程

1. 先向用户询问项目名。
2. 读取 `02.work session/session-4/ai-model-result/<项目名>.md`，列出所有对话轮次，向用户确认要分析第几次对话。
3. 从该文件中提取"用户第 N 次提示词"作为 prompt，提取"模型第 N 次回答内容"作为执行结果。
4. 读取 `02.work session/session-4/gitlab source/<项目名>/`，确认实际代码和结果状态。
5. 对照 prompt 要求与实际结果，判断任务完成度。
6. 判断这次结果整体是否满意。
7. 若为未完成，撰写未完成说明。
8. 若为不满意，撰写不满意的点。
9. 调用 `humanizer-zh.agent.md` 对未完成说明做去 AI 味润色。
10. 撰写过程满意度。
11. 将结果写入评价结果文件。

## 注意事项

1. 不要只根据模型自述判断是否完成，必须结合代码和结果文件一起看。
2. 不要把“没做成”和“做得不够好”混成一类。
3. 不要写太泛的总结，尽量落到实际业务、实际功能和实际验证结果。
4. 这个技能只做分析，不负责补代码、不负责修复、不负责继续实现。
